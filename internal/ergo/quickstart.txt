{{BOLD}}ergo quickstart{{RESET}}
{{CYAN}}Complete command reference for agents and humans.{{RESET}}

{{HEADER}}1. INITIALIZE AND FIND THE GRAPH{{RESET}}

  {{CMD}}ergo init{{RESET}}
  {{CMD}}ergo init path/to/repo{{RESET}}
  {{CMD}}ergo where{{RESET}}

`init` creates `.ergo/plans.jsonl` and `.ergo/lock`. Commands discover `.ergo`
by walking upward from the current directory or `--dir <path>`. Existing
repositories that use `.ergo/events.jsonl` remain supported.

{{HEADER}}2. INSPECT WORK{{RESET}}

  {{CMD}}ergo list{{RESET}}
  {{CMD}}ergo list --ready{{RESET}}
  {{CMD}}ergo list --all{{RESET}}
  {{CMD}}ergo list --epic OFKSTE{{RESET}}
  {{CMD}}ergo --json list{{RESET}}
  {{CMD}}ergo --json show ABCDEF{{RESET}}

Default list output shows active work. `--ready` shows only claimable todo leaf
tasks. `--all` also shows done and canceled work. `--epic` scopes the view to
one container. `show` returns full task details, results, dependencies, or a
container with its children.

Readiness is derived:

  todo + every dependency complete = ready

A todo task with unmet dependencies is waiting, not explicitly blocked. Use
`block` only when a person or agent has identified an impediment.

{{HEADER}}3. CLAIM AND EXECUTE{{RESET}}

Use a stable identity such as `model@hostname`.

Claim the oldest ready task:
  {{CMD}}ergo --json claim --agent sonnet@agent-host{{RESET}}

Claim or resume a specific task:
  {{CMD}}ergo --json claim ABCDEF --agent sonnet@agent-host{{RESET}}

Specific claim accepts todo, blocked, done, canceled, and legacy error leaf
tasks. It may resume a task even when automatic readiness would not select it.
A repeated claim by the same agent is a no-op. A different agent gets a claim
conflict. Claim and doing are one atomic write.

Claim JSON includes exact commands for every exit:

  "next_commands": {
    "done": "ergo --json done ABCDEF",
    "block": "ergo --json block ABCDEF",
    "cancel": "ergo --json cancel ABCDEF",
    "release": "ergo --json release ABCDEF"
  }

Oldest-ready claim returns success when no work is ready. JSON output is:
  {"status":"no_ready","message":"No ready ergo tasks."}

{{HEADER}}4. LEAVE OR CLASSIFY WORK{{RESET}}

  {{CMD}}ergo done ABCDEF{{RESET}}
  {{CMD}}ergo block ABCDEF{{RESET}}
  {{CMD}}ergo cancel ABCDEF{{RESET}}
  {{CMD}}ergo release ABCDEF{{RESET}}

`done`, `block`, and `cancel` accept any leaf-task state and establish their
named state. `release` accepts todo, doing, blocked, or legacy error and returns
the task to todo. release rejects done and canceled; use a specific claim when
closed work resumes. Every lifecycle exit clears the claim.

Use `release` when the objective remains valid and another attempt may proceed.
Use `block` when an identified impediment must be resolved first.

Repeating a command that already holds its postcondition succeeds without a
redundant state event. A repeated command may still attach a result or body.

{{HEADER}}5. BODY NOTES AND RESULTS{{RESET}}

Replace the body while finishing:

  {{CMD}}printf '%s\n' '## Completion' '- Implemented and verified.' | ergo done ABCDEF{{RESET}}

Attach a result:

  {{CMD}}ergo done ABCDEF --result docs/spec.md{{RESET}}
  {{CMD}}ergo block ABCDEF --result .scratch/repro.md --summary "Reproduction evidence"{{RESET}}
  {{CMD}}ergo release ABCDEF --result .scratch/attempt.md --summary "Retry evidence"{{RESET}}

Lifecycle stdin is optional. When piped, it replaces the body in the same
locked event batch as the result, state, and claim changes. TTY stdin leaves the
body unchanged.

`--result` names one existing project-relative file. It cannot escape the
project, name a directory, or point inside `.ergo/`. `--summary` is optional and
requires `--result`; the path is the default summary. Results retain their
attachment-time hash, modification time, and git commit when available.

Attach a late result by repeating the current lifecycle command:

  {{CMD}}ergo done ABCDEF --result docs/followup.md{{RESET}}

{{HEADER}}6. CREATE TASKS{{RESET}}

  {{CMD}}ergo new task '{"title":"Login"}'{{RESET}}
  {{CMD}}printf '%s\n' '## Goal' '- Implement signup' | ergo new task '{"title":"Signup"}'{{RESET}}
  {{CMD}}ergo new task '{"title":"Session tokens","epic":"OFKSTE"}'{{RESET}}
  {{CMD}}ergo --json new task '{"title":"Fix CVE","claim":"sonnet@agent-host"}'{{RESET}}
  {{CMD}}ergo new task '{"title":"Document decision","state":"done"}'{{RESET}}

The optional JSON object accepts `title`, `epic`, `state`, `claim`, and `result`.
Title is required. State is todo, doing, blocked, done, or canceled. Claim
implies doing; doing otherwise requires `--agent`. New commands never create
the legacy error state. Piped stdin is the literal body, including an empty
pipe for an empty body.

Assigning the first child promotes a root task to a container only when it is
todo, unclaimed, and has no results.

{{HEADER}}7. EDIT CONTENT AND PLACEMENT{{RESET}}

  {{CMD}}ergo title ABCDEF "Clarify authentication failure"{{RESET}}
  {{CMD}}printf '%s\n' '## Goal' '- Clarify the failure' | ergo body ABCDEF{{RESET}}
  {{CMD}}printf '' | ergo body ABCDEF{{RESET}}
  {{CMD}}ergo move ABCDEF OFKSTE{{RESET}}
  {{CMD}}ergo move ABCDEF --root{{RESET}}

Title rejects blank text. Body requires piped stdin and treats it literally.
Both work on leaf tasks and containers.

Move accepts leaf tasks only. The destination must be a root container or a
clean root todo task that can be promoted. Containers cannot move or nest.
Self-moves and moves that create a container-child dependency are rejected.
Moving to the current parent is a no-op.

{{HEADER}}8. CREATE A CONTAINER PLAN{{RESET}}

  {{CMD}}ergo --json plan --file tasks.md '{"title":"Auth system"}'{{RESET}}

Example `tasks.md`:

  # Schema
  Create tables and indexes.
  ---
  # Endpoints
  Add signup and login handlers.

The file is split on a line that is exactly `---`. Each chunk starts with
`# Title`; the remaining markdown becomes the child body. The JSON argument
requires the container `title`. File order does not create dependencies.

{{HEADER}}9. DEPENDENCIES{{RESET}}

  {{CMD}}ergo sequence TASK_A TASK_B{{RESET}}
  {{CMD}}ergo sequence TASK_A TASK_B TASK_C{{RESET}}
  {{CMD}}ergo sequence rm TASK_A TASK_B{{RESET}}

The first command makes B depend on A. A chain makes each task depend on the
previous task. `sequence rm A B` removes the edge where B depends on A.

Self-dependencies, cycles, and dependencies between a container and its own
child are forbidden. A dependency on a container is complete when every child
is done or canceled.

{{HEADER}}10. JSON AND ERRORS{{RESET}}

Global flags:

  {{CMD}}ergo --json ...{{RESET}}
  {{CMD}}ergo --agent model@host ...{{RESET}}
  {{CMD}}ergo --dir path/to/repo ...{{RESET}}

Successful `--json` commands write exactly one JSON value to stdout. Direct
mutations return `kind`, `id`, `updated_fields`, and current state. A true no-op
has an empty `updated_fields` array. `claimed_by` is present when the task is
doing. Result, body, state, and claim changes share one atomic event batch.

Failures return nonzero and write an actionable error to stderr. A missing
graph suggests both `ergo init` and `ergo --dir <path>`. Removed v1 mutation
syntax points to the direct v2 commands.

{{HEADER}}11. PRUNE, COMPACT, AND LEGACY PLANS{{RESET}}

  {{CMD}}ergo prune{{RESET}}
  {{CMD}}ergo prune --yes{{RESET}}
  {{CMD}}ergo compact{{RESET}}

Prune previews tombstones by default. `--yes` tombstones done and canceled leaf
tasks and then empty containers. Compact rewrites the event log to the current
live graph and physically removes pruned history.

Legacy `.ergo/events.jsonl`, error states, and claimed-blocked tasks remain
readable. Opening a repository never rewrites it. Compact preserves unresolved
legacy state and ownership exactly. A direct lifecycle command normalizes only
the task it addresses. Older Ergo binaries may append events; v2 replays them
and normalizes them on the next explicit lifecycle command.

{{HEADER}}12. INVARIANTS{{RESET}}

  - Forward states are todo, doing, blocked, done, and canceled.
  - A claim exists if and only if state is doing.
  - Containers have children and no direct claim, lifecycle, or result behavior.
  - Containers complete when every child is done or canceled.
  - Writes and coherent reads acquire `.ergo/lock`.
  - Oldest-ready selection and claim are atomic under that lock.
  - `version` or `--version` prints the build version.
